PART 1

Data Retrieval — JOINs

JOIN Task 1 — INNER JOIN TASK 01 
SELECT e.employee_name, e.salary, d.department_name
FROM employees e
INNER JOIN departments d
  ON e.department_id = d.department_id;
How to get this: Use INNER JOIN when you only want rows where both tables have a match. Employees without a department, and departments with no employees, are both excluded. Think of it as the intersection of two sets.
JOIN Task 2 — LEFT JOIN: All employees TASK 02 
SELECT e.employee_name, e.salary, d.department_name
FROM employees e
LEFT JOIN departments d
  ON e.department_id = d.department_id;
How to get this: LEFT JOIN keeps ALL rows from the LEFT table (employees). If an employee has no matching department, department_name will be NULL. The left table is always "preserved" completely.
JOIN Task 3 — RIGHT JOIN: All departments TASK 03 
SELECT e.employee_name, e.salary, d.department_name
FROM employees e
RIGHT JOIN departments d
  ON e.department_id = d.department_id;
How to get this: RIGHT JOIN keeps ALL rows from the RIGHT table (departments). Departments that have no employees show NULL for all employee columns. Use this to audit empty departments.
PART 2

Transaction Processing

Transaction Task 4 — Salary Transfer with COMMIT TASK 04 
START TRANSACTION;

UPDATE employees
  SET salary = salary - 5000
  WHERE employee_id = 1;

UPDATE employees
  SET salary = salary + 5000
  WHERE employee_id = 2;

COMMIT;
How to get this: START TRANSACTION opens a transaction block. All changes inside are temporary until COMMIT permanently saves them. This ensures both the deduction and addition happen together — atomically.
Transaction Task 5 — Failed Transaction with ROLLBACK TASK 05 
START TRANSACTION;

UPDATE employees
  SET salary = salary - 5000
  WHERE employee_id = 1;

-- Error detected here: cancel everything
ROLLBACK;
How to get this: ROLLBACK undoes every change made since START TRANSACTION. The salary deduction is reversed completely. The database looks exactly as it did before the transaction started.
PART 3

Stored Procedures

Procedure Task 6 — Create Stored Procedure TASK 06 
DELIMITER $$

CREATE PROCEDURE transfer_salary(
  IN sender_id INT,
  IN receiver_id INT,
  IN amount DECIMAL(10,2)
)
BEGIN
  START TRANSACTION;

    UPDATE employees
      SET salary = salary - amount
      WHERE employee_id = sender_id;

    UPDATE employees
      SET salary = salary + amount
      WHERE employee_id = receiver_id;

  COMMIT;
END$$

DELIMITER ;
How to get this: DELIMITER $$ prevents semicolons inside the procedure from ending the statement early. Parameters tagged IN accept values when called. The transaction inside ensures both UPDATEs are atomic.
Procedure Task 7 — Execute Procedure TASK 07 
CALL transfer_salary(1, 2, 5000.00);
How to get this: Use CALL procedure_name(args) to execute a stored procedure. Here: employee 1 (sender) transfers ₱5,000 to employee 2 (receiver). The procedure handles all the transaction logic internally.
PART 4

Stored Function

Function Task 8 — Total Salary per Department TASK 08 
DELIMITER $$

CREATE FUNCTION get_dept_total_salary(dept_id INT)
RETURNS DECIMAL(15,2)
DETERMINISTIC
BEGIN
  DECLARE total DECIMAL(15,2);

  SELECT SUM(salary) INTO total
  FROM employees
  WHERE department_id = dept_id;

  RETURN total;
END$$

DELIMITER ;

-- Usage:
SELECT get_dept_total_salary(1);
How to get this: Functions use RETURNS datatype and end with RETURN value. DECLARE creates a local variable. SELECT ... INTO variable stores the query result. Unlike procedures, functions can be embedded directly in SELECT statements.

Critical Thinking — Improved Procedure

Advanced Task 9 — Safe Transfer (Prevent Negative Salary) TASK 09 
DELIMITER $$

CREATE PROCEDURE safe_transfer(
  IN sender_id INT,
  IN receiver_id INT,
  IN amount DECIMAL(10,2)
)
BEGIN
  DECLARE sender_salary DECIMAL(10,2);

  SELECT salary INTO sender_salary
  FROM employees
  WHERE employee_id = sender_id;

  IF sender_salary >= amount THEN
    START TRANSACTION;
      UPDATE employees
        SET salary = salary - amount
        WHERE employee_id = sender_id;

      UPDATE employees
        SET salary = salary + amount
        WHERE employee_id = receiver_id;
    COMMIT;
  ELSE
    ROLLBACK;
    SELECT 'Error: Insufficient salary balance.' AS message;
  END IF;
END$$

DELIMITER ;
How to get this: Before transferring, SELECT ... INTO sender_salary reads the current balance. The IF block checks if the sender has enough. If not, ELSE triggers ROLLBACK and returns an error message — preventing any negative salary situation.
